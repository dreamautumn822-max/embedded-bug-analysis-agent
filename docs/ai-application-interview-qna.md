# AI 应用开发岗位面试问答准备

本文档面向 AI 应用开发岗位面试，回答会尽量结合当前项目：

> 嵌入式网通设备 Bug 分析 Agent：输入设备型号、固件版本、故障现象和设备日志，通过 LangGraph 编排日志解析、历史 Bug 检索、模块文档检索、代码线索检索，输出 Bug 类型、可能根因、证据链和修复建议。

## 1. 自我介绍与转型动机

### 1.1 你为什么从嵌入式开发转 AI 应用开发？

**回答思路**

- 不要说“AI 火，所以想转”。
- 要强调你的嵌入式背景是优势：懂真实业务场景、日志、故障定位、设备问题。
- AI 应用开发不是纯调模型，而是把模型能力接入业务流程。

**参考回答**

我之前做的是网通设备相关的嵌入式故障检测项目，接触比较多的是路由器、光猫这类设备的日志分析、异常定位和故障归因。后来我发现很多研发和测试排障工作其实有明显的流程：先看现象，再看日志，再关联历史 Bug、模块文档和代码修改点。

所以我希望往 AI 应用开发方向发展，不是单纯做聊天机器人，而是把 LLM、RAG 和 Agent 用在真实工程场景里。我的项目就是基于这个思路做的：用 LangGraph 把嵌入式网通设备 Bug 分析流程编排成一个 Agent，让系统根据日志、历史缺陷、文档和代码线索生成根因分析和修复建议。

### 1.2 你觉得你的嵌入式背景对 AI 应用开发有什么帮助？

**参考回答**

我觉得主要有三点帮助。

第一，AI 应用最终要落在业务场景里。我做过嵌入式网通设备，对设备日志、网络模块、固件版本、DHCP、PPPoE、Wi-Fi 这类故障场景比较熟，能设计出更贴近真实用户的 AI 应用。

第二，嵌入式开发比较重视稳定性、状态机、异常处理和可观测性，这些在 Agent 应用里也很重要。Agent 不能只是能跑一次，还要能追踪每一步状态、处理失败、避免乱调用工具。

第三，我有工程化经验，能把模型能力封装成 API、页面和评估脚本，而不是停留在 prompt demo。

## 2. LLM 基础

### 2.1 Transformer 为什么适合大语言模型？

**参考回答**

Transformer 的核心是 self-attention，它可以让模型在处理一个 token 时关注上下文中其他 token 的信息。相比传统 RNN，Transformer 更适合并行训练，也更容易扩展到大规模参数和大规模语料。

在大语言模型里，Transformer 通过预训练学习语言规律、知识关联和推理模式。推理时，模型根据已有上下文预测下一个 token，从而生成连续文本。

### 2.2 Token 是什么？为什么上下文窗口重要？

**参考回答**

Token 可以理解为模型处理文本的基本单位，可能是一个字、一个词，也可能是词的一部分。模型不是直接按字符或句子理解文本，而是把输入切成 token 后处理。

上下文窗口决定了一次请求中模型最多能看到多少 token。对于 AI 应用来说，上下文窗口很重要，因为 RAG 检索结果、用户问题、系统提示词、历史对话都会占用 token。如果日志或文档太长，就需要做摘要、切分、检索和压缩，不能直接全部塞给模型。

### 2.3 Temperature 和 top_p 是什么？

**参考回答**

Temperature 控制生成的随机性。值越低，输出越稳定、确定；值越高，输出越发散、创造性更强。

top_p 是 nucleus sampling，它限制模型只在累计概率达到 p 的候选 token 里采样。

在 Bug 分析、客服、知识库问答这类场景里，我一般会使用较低 temperature，比如 0 到 0.3，因为更需要稳定、可复现和基于证据的输出，不需要太强的创造性。

### 2.4 为什么大模型会幻觉？怎么降低幻觉？

**参考回答**

大模型的本质是根据上下文生成概率最高的文本，它不天然知道哪些内容是真实的，也不会自动区分“知道”和“不知道”。当上下文缺少事实依据、问题超出模型知识范围，或者 prompt 要求它必须回答时，就容易产生幻觉。

降低幻觉可以从几个方面做：

- 使用 RAG，把答案限制在检索到的资料范围内。
- 要求模型引用证据，输出来源。
- 对模型输出做结构化校验。
- 对高风险问题设置“不知道”或人工确认。
- 降低 temperature。
- 对工具调用结果和最终答案做一致性检查。

在我的项目里，根因分析不会只让模型凭空生成，而是结合日志、历史 Bug、模块文档和代码线索，并把这些证据展示在页面的证据总线里。

### 2.5 微调、RAG、Prompt 优化分别适合什么场景？

**参考回答**

Prompt 优化适合快速调整任务说明、输出格式和推理步骤，成本低，适合早期验证。

RAG 适合解决私有知识、知识更新、文档问答和可追溯回答问题。比如企业内部文档、历史 Bug、代码片段，这些不适合全部放进模型参数里，更适合检索后放进上下文。

微调更适合让模型学习稳定的输出风格、固定任务模式或领域表达习惯，但它不适合频繁更新知识。如果知识经常变化，优先考虑 RAG。

## 3. RAG 高频问题

### 3.1 RAG 的完整流程是什么？

**参考回答**

RAG 一般分为离线和在线两部分。

离线阶段：

1. 收集文档、代码、FAQ、历史案例等知识源。
2. 清洗文档。
3. 切分 chunk。
4. 生成 embedding。
5. 写入向量数据库或检索索引。

在线阶段：

1. 用户提出问题。
2. 对问题做改写或提取关键词。
3. 到向量库或关键词索引中检索相关 chunk。
4. 对检索结果做 rerank 或过滤。
5. 把问题和相关上下文交给 LLM。
6. 生成带证据的回答。
7. 对输出做格式校验和质量评估。

### 3.2 文档怎么切分 chunk？

**参考回答**

chunk 切分要根据文档类型决定。

普通说明文档可以按标题、段落、语义块切分。代码文档可以按函数、类、文件结构切分。日志类数据可以按时间窗口、会话、错误上下文切分。

chunk 太小会丢上下文，chunk 太大又会降低检索精度并浪费 token。所以一般会结合业务测试集调参，比如先从 300 到 800 tokens 开始尝试，再看召回效果。

对于我的项目，如果是模块文档，我会按小标题切分；如果是代码，我更倾向按函数或相关代码块切分；如果是设备日志，我会围绕错误行保留前后若干行上下文。

### 3.3 chunk size 和 overlap 怎么选？

**参考回答**

chunk size 主要影响上下文完整性和检索精度。size 大，信息更完整，但可能包含噪声；size 小，检索更精确，但容易缺少上下文。

overlap 是为了避免重要信息刚好被切分边界截断。比如一个故障描述跨两个段落，适当 overlap 可以让两个 chunk 都保留部分上下文。

我的做法是先用经验值，比如 500 tokens 左右、10% 到 20% overlap，然后用评估集看正确文档是否能被召回，再调整参数。

### 3.4 向量检索和关键词检索有什么区别？

**参考回答**

关键词检索更适合精确匹配，比如错误码、函数名、Bug ID、日志关键字。向量检索更适合理解语义相近但字面不同的内容。

在工程实践中，最好不要只用一种。比如嵌入式日志里有 `lease allocation failed`、`pppoe auth failed`、`BUG-018` 这种强关键词，关键词检索很有价值；但用户描述可能是“升级后客户端拿不到 IP”，这时向量检索可以召回 DHCP 相关资料。

所以更稳妥的是混合检索：向量召回语义相关内容，关键词保证关键错误信息不丢失。

### 3.5 TopK 怎么选？

**参考回答**

TopK 不是越大越好。TopK 太小可能漏掉关键资料，TopK 太大又会引入噪声，增加 token 成本，甚至干扰模型判断。

一般会根据任务复杂度和上下文窗口选择。简单 FAQ 可能 TopK=3 就够；复杂故障分析可以取更多，但需要 rerank 或按来源分组。

在我的项目里，历史 Bug、模块文档、代码线索可以分别取 TopK，比如每类取 2 到 3 条，而不是所有资料混在一起取一个 TopK。这样证据来源更清晰。

### 3.6 如果检索不到正确资料怎么办？

**参考回答**

我会从四层排查。

第一，知识库是否有正确资料。如果没有，需要补充数据。

第二，文档切分是否合理。如果答案被切碎或上下文丢失，要调整 chunk。

第三，检索 query 是否合理。可以对用户问题做 query rewrite，提取关键词、错误码、模块名。

第四，检索方式是否单一。对于代码、日志、错误码，应该结合关键词检索和向量检索。

如果仍然检索不到，最终回答应该明确提示证据不足，而不是让模型硬猜。

### 3.7 怎么评估 RAG 效果？

**参考回答**

我会分三层评估。

第一是检索层：正确文档是否被召回，关注 recall@k、hit rate、MRR。

第二是生成层：答案是否基于证据，是否引用正确来源，是否出现幻觉。

第三是任务层：用户能不能根据答案完成任务，比如定位 Bug 根因、找到相关代码、形成修复建议。

对于我的项目，可以用历史 Bug case 做评估集。每个 case 有设备现象、日志、标准 Bug 类型、标准根因和应召回证据，然后评估 bug_type 准确率、证据覆盖率和根因关键词命中率。

## 4. LangChain / LangGraph

### 4.1 LangChain 主要解决什么问题？

**参考回答**

LangChain 是一个 LLM 应用开发框架，主要提供模型调用、Prompt 模板、Retriever、Tool、Chain、Agent、Memory 等组件，帮助开发者更快地搭建 LLM 应用。

它的价值不是让模型更聪明，而是把模型和外部系统连接起来，比如知识库、数据库、搜索工具、业务 API。

### 4.2 LangGraph 和 LangChain 是什么关系？

**参考回答**

LangGraph 可以理解为 LangChain 生态里更适合构建有状态、多步骤 Agent 的框架。LangChain 更偏组件和链式调用，LangGraph 更强调图结构、状态流和可控执行。

在复杂 Agent 里，不同步骤之间有状态传递，也可能有条件分支、循环、失败重试。LangGraph 用节点和边来组织这些流程，比简单 Chain 更清晰。

### 4.3 你的项目中 LangGraph 怎么用？

**参考回答**

我的项目把 Bug 分析流程拆成多个节点：

1. 提取故障信息。
2. 解析设备日志。
3. 检索历史 Bug。
4. 检索模块文档。
5. 检索代码线索。
6. 生成根因假设。
7. 生成最终报告。

这些节点共享一个状态对象，比如设备型号、固件版本、日志解析结果、相关 Bug、相关文档、相关代码和最终报告。LangGraph 负责让这些节点按顺序执行，并把每一步的结果写回状态。

这样做的好处是流程清晰、可调试、可测试，也方便后续加节点，比如接入真实 LLM、加 rerank、加人工确认。

### 4.4 为什么 Agent 项目要用状态流？

**参考回答**

Agent 不是一次简单函数调用，它通常包含多个步骤：理解任务、调用工具、观察结果、继续推理、生成答案。每一步都会产生中间结果。

状态流可以把这些中间结果显式保存下来，比如解析出的错误模式、检索到的文档、工具调用结果、当前置信度等。这样系统更容易调试，也更容易做失败恢复和日志追踪。

如果没有状态流，所有信息都混在 prompt 或局部变量里，后期会很难维护。

### 4.5 什么时候不用 Agent，只用普通 Chain？

**参考回答**

如果任务流程固定、步骤简单、不需要动态选择工具，就不一定要用 Agent。比如单轮文档问答、固定格式摘要、简单分类，用普通 Chain 就够。

Agent 更适合任务不确定、需要多工具协作、需要根据中间结果动态决策的场景。

我的 Bug 分析项目虽然目前流程比较固定，但它天然适合扩展成 Agent，因为后续可以根据不同 bug_type 决定检索哪个模块、调用哪个诊断工具、是否要求补充日志。

## 5. Agent 设计

### 5.1 什么是 AI Agent？

**参考回答**

AI Agent 是能围绕一个目标进行多步骤决策和执行的系统。它通常包含模型、工具、记忆或状态、执行流程和反馈机制。

普通 ChatBot 主要是根据用户输入生成回复，而 Agent 可以根据任务需要调用工具、检索资料、执行代码、查询数据库，然后根据观察结果继续下一步。

### 5.2 Agent 和普通 ChatBot 有什么区别？

**参考回答**

普通 ChatBot 更像问答系统，输入一句话，输出一句话。Agent 更像任务执行系统，它会拆解任务、维护状态、调用工具、观察结果，再继续推理。

比如普通 ChatBot 可以回答“DHCP 获取不到 IP 可能是什么原因”。但我的 Bug 分析 Agent 会解析日志、检索历史 Bug、查模块文档、找代码线索，然后输出带证据链的根因分析。

### 5.3 Agent 如何避免乱调用工具？

**参考回答**

我会从几个方面限制。

第一，明确工具描述，让模型知道每个工具适合什么输入和输出。

第二，限制工具集合，不给 Agent 暴露无关或高风险工具。

第三，使用状态机或 LangGraph 控制流程，不完全让模型自由决定所有步骤。

第四，对工具参数做 schema 校验。

第五，对高风险动作加人工确认，比如删除数据、执行命令、修改配置。

### 5.4 Agent 出错怎么恢复？

**参考回答**

要先区分错误类型。模型输出格式错，可以做结构化解析重试；工具调用超时，可以设置 timeout 和 retry；检索为空，可以降级到关键词检索或提示用户补充信息；关键步骤失败，可以中断流程并返回明确错误。

工程上还要记录每个节点的输入、输出、耗时和异常。这样出问题时可以定位是哪一步失败，而不是只看到最终答案错误。

### 5.5 ReAct 是什么？

**参考回答**

ReAct 是 Reasoning and Acting 的组合。模型不是一次性给出答案，而是交替进行思考和行动：

- Thought：分析下一步该做什么。
- Action：调用工具。
- Observation：观察工具结果。
- 再继续 Thought，直到得到最终答案。

它适合需要多步工具调用的问题。但生产系统里通常不会把自由 ReAct 完全放开，而是会加工具限制、状态控制和输出校验。

## 6. 工程落地

### 6.1 AI 应用怎么设计 API？

**参考回答**

我会把 AI 能力封装成明确的后端接口。以我的项目为例，`/analyze` 接口输入设备型号、固件版本、问题现象、日志、可选堆栈和模块提示，输出 Bug 类型、摘要、根因、证据、修复建议和置信度。

API 层负责参数校验、调用 Agent 流程、返回结构化结果。这样前端、测试脚本和其他系统都可以稳定接入。

### 6.2 怎么处理模型返回格式不稳定？

**参考回答**

首先在 prompt 里明确输出 JSON schema 或字段要求。其次使用 Pydantic 这类工具做结构化校验。如果校验失败，可以让模型基于错误信息重试一次。

对于关键业务字段，比如 bug_type、confidence、evidence，不能只靠自然语言解析，应该有明确枚举、范围和默认降级策略。

### 6.3 怎么处理超时和失败？

**参考回答**

模型调用和外部工具调用都要设置 timeout，不能无限等待。失败时可以根据场景做 retry、fallback 或返回可解释错误。

比如 LLM 调用失败，可以返回“模型服务暂不可用”；检索失败，可以提示“知识库检索失败”；部分证据为空时，可以继续生成报告，但要标注证据不足。

### 6.4 怎么控制成本？

**参考回答**

成本主要来自模型 token 和调用次数。可以从几方面控制：

- 使用更小模型处理简单任务。
- 对检索结果做压缩，只把必要 chunk 放进上下文。
- 缓存 embedding、检索结果和常见问题答案。
- 减少无效 Agent 循环。
- 对长日志先做规则提取或摘要，再交给模型。

### 6.5 怎么做日志和可观测性？

**参考回答**

AI 应用要记录的不只是接口日志，还要记录模型相关信息。

我会记录请求 ID、用户输入摘要、检索 query、召回文档 ID、模型名称、prompt 版本、输出结果、耗时、token 用量、错误信息和每个 Agent 节点状态。

这样后续才能定位问题，比如是检索没召回，还是模型没有正确使用证据，还是工具调用失败。

## 7. 项目深挖

### 7.1 你的项目解决什么问题？

**参考回答**

我的项目解决的是嵌入式网通设备故障分析效率问题。像路由器、光猫这类设备出现 DHCP 获取不到 IP、PPPoE 拨号失败、Wi-Fi 断连等问题时，研发或测试需要查看设备日志、历史 Bug、模块文档和代码线索，才能判断根因。

我把这个流程做成了一个 Bug 分析 Agent。用户输入设备型号、固件版本、故障现象和日志，系统输出 Bug 类型、可能根因、证据链和修复建议。

### 7.2 为什么选 Bug 分析，而不是普通知识库问答？

**参考回答**

普通知识库问答更偏“问一个问题，找一段文档回答”。但 Bug 分析更接近真实研发工作流，它不是只查文档，还要结合日志、历史缺陷、代码片段和模块知识做判断。

这个项目能体现 AI 应用开发里的几个关键能力：RAG、Agent 状态流、工具检索、结构化输出、前后端接口和评估。这比单纯做一个聊天机器人更有业务含义。

### 7.3 你的 Agent 流程是什么？

**参考回答**

流程是：

1. 用户提交设备型号、固件版本、问题现象和日志。
2. 系统解析日志，提取错误模式和关键事件。
3. 根据问题现象和日志关键词判断 bug_type。
4. 检索历史 Bug，找相似问题。
5. 检索模块文档，找相关背景知识。
6. 检索代码线索，找可能相关函数或文件。
7. 生成根因假设、证据链和修复建议。
8. 前端按 LOG、DOC、BUG、CODE 展示证据总线。

### 7.4 你的项目和普通 ChatBot 有什么区别？

**参考回答**

普通 ChatBot 主要依赖模型直接回答，容易出现泛泛而谈或者幻觉。我的项目是一个面向具体工程场景的 Agent 应用，特点是有明确输入输出、有状态流、有本地知识检索、有证据链、有结构化结果。

它不是只回答“可能是 DHCP 问题”，而是会展示日志证据、历史 Bug、模块文档和代码线索，让用户能复查为什么得到这个结论。

### 7.5 当前项目有没有接真实大模型？

**参考回答**

当前版本主要是本地规则链和检索流程，用来验证 Agent/RAG 的工程架构。它已经具备 LangGraph 状态流、FastAPI 接口、Streamlit 前端、RAG 数据源和评估脚本。

如果接真实大模型，我会优先替换根因生成和报告生成部分，让 LLM 基于检索到的证据生成更自然、更完整的分析报告。同时保留现有结构化输出和证据校验，避免模型自由发挥。

### 7.6 如果接入真实大模型，你会怎么改？

**参考回答**

我会分三步改。

第一，把 `extract_chain`、`root_cause_chain`、`report_chain` 抽象成 LLM 调用层，接入 OpenAI、DeepSeek 或 Qwen 这类模型。

第二，使用结构化输出，要求模型返回固定 JSON，包括 bug_type、root_causes、evidence_refs、fix_suggestions、confidence。

第三，增加防幻觉约束：模型只能基于检索到的日志、文档、历史 Bug 和代码证据回答。如果证据不足，要明确返回“证据不足，需要补充日志”。

### 7.7 这个项目最大难点是什么？

**参考回答**

我认为难点不是把模型接进来，而是把故障分析流程拆成稳定、可解释、可测试的步骤。

Bug 分析本身涉及日志、模块知识、历史问题和代码线索。如果所有内容直接塞给模型，结果不可控，也很难调试。所以我用 LangGraph 把流程拆成多个节点，每一步都有输入输出和状态记录。这样即使结果不准，也能定位是日志解析问题、检索问题，还是生成问题。

## 8. 评估与优化

### 8.1 AI 应用怎么评估？

**参考回答**

AI 应用评估不能只看回答“像不像”。我会从任务目标出发设计指标。

对于 Bug 分析项目，可以评估：

- bug_type 分类准确率。
- 正确历史 Bug 是否被召回。
- 文档和代码证据是否相关。
- 根因描述是否命中标准答案关键词。
- 修复建议是否可执行。
- 输出是否引用了证据。

同时需要人工评估一部分 case，因为根因分析有时不是单一标准答案。

### 8.2 怎么减少 RAG 幻觉？

**参考回答**

首先要让模型知道答案必须基于检索结果。其次要把证据和回答绑定起来，比如每个根因都要对应日志、文档或历史 Bug 证据。

如果检索结果为空或相关性低，系统应该返回证据不足，而不是让模型自由猜。还可以在生成后做一致性检查，看答案中的关键结论是否能在证据中找到来源。

### 8.3 如果日志很长，超过上下文窗口怎么办？

**参考回答**

不能直接把所有日志塞进模型。我会先做日志预处理：

- 按时间、模块、错误级别过滤。
- 提取 error、warning、关键事件。
- 保留错误行前后上下文。
- 对重复日志去重。
- 必要时先做分段摘要。

对于嵌入式设备日志，很多关键问题集中在少数错误行和状态变化附近，比如 link down/up、DHCP failure、PPPoE auth failed，所以可以先用规则提取，再交给 LLM 分析。

### 8.4 怎么判断回答质量？

**参考回答**

我会看五点：

1. 是否回答了用户问题。
2. 是否基于证据。
3. 是否有明确结论，而不是泛泛而谈。
4. 是否给出可执行建议。
5. 是否在证据不足时明确说明限制。

对于我的项目，好的输出应该包含 Bug 类型、根因假设、日志证据、历史 Bug 证据、代码线索和修复建议。

## 9. Python / 后端基础

### 9.1 FastAPI 在项目里做什么？

**参考回答**

FastAPI 负责把 Agent 能力封装成 HTTP 服务。我的项目里有 `/health` 用于健康检查，`/analyze` 用于提交 Bug 分析请求。

请求体用 Pydantic schema 校验，保证必须包含设备型号、固件版本、问题现象和日志。返回也是结构化 schema，方便前端展示和测试。

### 9.2 Pydantic 有什么作用？

**参考回答**

Pydantic 用来做数据校验和结构化建模。在 AI 应用里，它可以保证 API 输入输出符合预期，也可以用来校验模型返回的 JSON。

比如模型返回 confidence，如果不是数字或者超出范围，就可以被 schema 捕获，而不是错误数据继续流入后续系统。

### 9.3 为什么要写测试？

**参考回答**

AI 应用虽然有模型的不确定性，但工程部分更需要测试。比如 API schema、日志解析、检索逻辑、状态流、证据分组、前端 payload 组装都应该可测试。

我的项目里有单元测试和评估脚本。这样改前端或调整 Agent 节点时，可以确保基础功能没有回归。

### 9.4 如果线上分析结果不准，你怎么排查？

**参考回答**

我会沿着 Agent 流程逐步排查。

第一，看输入日志是否足够。

第二，看日志解析是否提取到正确错误模式。

第三，看检索 query 是否合理。

第四，看历史 Bug、文档、代码是否召回正确证据。

第五，看模型是否正确使用证据生成结论。

如果是检索问题，就优化 chunk、query rewrite 或 rerank；如果是生成问题，就调整 prompt、结构化输出或增加证据约束。

## 10. 十个重点问题速背版

### 10.1 你为什么从嵌入式开发转 AI 应用开发？

我不是完全脱离原来的方向，而是把嵌入式网通设备的故障分析经验和 AI 应用结合起来。AI 应用开发需要真实业务场景，我熟悉设备日志、故障定位和网络模块，所以选择做 Bug 分析 Agent，把日志解析、历史 Bug、文档和代码线索结合起来，提高排障效率。

### 10.2 你这个 Bug 分析 Agent 的完整流程是什么？

用户输入设备型号、固件版本、故障现象和日志。系统先解析日志和提取故障信息，然后检索历史 Bug、模块文档和代码线索，再生成 Bug 类型、根因假设、证据链和修复建议。前端会把证据按 LOG、DOC、BUG、CODE 分组展示，方便用户复查。

### 10.3 LangChain 和 LangGraph 在项目中分别做了什么？

LangChain 生态主要提供 LLM 应用组件的思路，比如链、检索器、工具和结构化调用。LangGraph 在项目里负责状态流编排，把日志解析、Bug 检索、文档检索、代码检索和报告生成拆成节点，每个节点读写共享状态，使流程更清晰、可调试。

### 10.4 RAG 在项目里解决了什么问题？

RAG 解决的是模型缺少项目私有知识的问题。网通设备 Bug 分析需要历史缺陷、模块文档和代码信息，这些不是通用模型天然知道的。通过 RAG，系统可以先检索相关资料，再基于证据生成分析结果，减少幻觉，提高可追溯性。

### 10.5 为什么 Bug 分析适合用 Agent？

Bug 分析不是单步问答，而是一个多步骤流程：理解现象、解析日志、查历史问题、看模块文档、找代码线索、形成根因和修复建议。Agent 适合这种需要多工具、多阶段推理和状态管理的任务。

### 10.6 如果接入真实大模型，你会怎么设计？

我会把根因生成和报告生成节点替换成真实 LLM 调用，比如 OpenAI、DeepSeek 或 Qwen。检索和日志解析仍然保留，模型只基于检索到的证据生成结构化 JSON。然后用 Pydantic 校验输出，并在证据不足时要求模型明确说明，而不是编造结论。

### 10.7 怎么减少模型幻觉？

我会用 RAG 提供事实依据，要求答案引用证据；用结构化输出限制模型格式；降低 temperature；在证据不足时允许回答“不确定”；并对最终结论和证据做一致性检查。我的项目中前端展示证据总线，就是为了让结论可复查。

### 10.8 怎么评估这个系统分析得准不准？

可以用历史 Bug case 做评估集。每条 case 包含输入日志、标准 bug_type、标准根因和应召回证据。指标包括 bug_type 准确率、证据召回率、根因关键词命中率、修复建议可执行性。部分复杂 case 还需要人工评审。

### 10.9 如果日志很长，超过上下文窗口怎么办？

先不要直接塞给模型，而是做日志预处理。按错误级别、模块、时间窗口过滤，提取关键 error 和状态变化，保留错误行前后上下文，去掉重复日志。必要时分段摘要，再把最关键的日志片段交给模型。

### 10.10 这个项目相比普通 ChatBot 有什么亮点？

它不是普通聊天页面，而是面向嵌入式网通设备故障分析的 Agent 应用。它有明确业务输入输出，有 LangGraph 状态流，有历史 Bug、模块文档和代码线索检索，有证据链展示和评估脚本。它更接近真实 AI 应用开发岗位需要的工程能力。
