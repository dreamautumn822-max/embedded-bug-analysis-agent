# AI 应用开发通用能力学习笔记

本文档用于理解简历中“AI 应用开发”技能区应该表达的通用能力。它不只针对某一个项目，而是面向 AI 应用开发岗位常见要求：LLM 调用、Prompt、RAG、Agent、LangChain/LangGraph、模型 API 接入、工程稳定性等。

## 1. LLM 应用开发基本流程

### 1.1 这项能力是什么意思

LLM 应用开发不是简单调用一个大模型接口，而是把用户输入、业务上下文、Prompt、模型调用、结果解析、异常处理和前后端系统串成一个完整应用。

一个典型 LLM 应用流程是：

```text
用户输入
  -> 输入校验
  -> 构造 Prompt / 上下文
  -> 调用大模型 API
  -> 解析模型输出
  -> 结构化校验
  -> 业务后处理
  -> 返回前端或 API 调用方
```

如果是复杂应用，中间还会加入 RAG 检索、工具调用、状态流、缓存、日志、权限控制和评估。

### 1.2 为什么重要

很多人以为 AI 应用就是“前端输入框 + 调模型 + 显示结果”。真实业务里不是这样。

生产环境里的 AI 应用需要解决：

- 用户输入不规范怎么办？
- 模型回答格式不稳定怎么办？
- 模型超时怎么办？
- 模型回答错了怎么排查？
- 模型生成内容如何被后端系统继续使用？
- 如何让回答基于业务资料，而不是模型自由发挥？

所以 AI 应用开发岗位看重的不只是“会调 API”，而是能不能把大模型能力封装成稳定、可维护的业务系统。

### 1.3 需要掌握哪些点

你需要理解这些基础概念：

- **Prompt**：给模型的任务说明和上下文。
- **System Prompt**：定义模型角色、边界和输出规则。
- **User Prompt**：用户真实输入。
- **Context**：额外业务信息，比如文档、历史记录、工具返回结果。
- **Model Parameters**：temperature、top_p、max_tokens 等控制项。
- **Structured Output**：让模型按 JSON 或固定 schema 输出。
- **Validation**：用 Pydantic 等工具检查模型输出是否符合要求。
- **Fallback**：模型失败或输出异常时的降级方案。

### 1.4 工程实践

一个好的 LLM 应用通常会做这些事：

1. **输入层校验**

   比如用户必须传入 `device_model`、`firmware_version`、`symptom`、`logs`，否则直接返回参数错误。

2. **Prompt 模板化**

   不要把 prompt 散落在代码里，而是固定模板，方便版本管理和调试。

3. **上下文构造**

   把用户问题、检索到的文档、历史 Bug、代码片段等按清晰结构传给模型。

4. **模型输出结构化**

   让模型返回固定字段，比如：

   ```json
   {
     "bug_type": "network_dhcp",
     "root_causes": [],
     "evidence": [],
     "fix_suggestions": [],
     "confidence": 0.92
   }
   ```

5. **结果校验**

   用 schema 检查字段是否存在、类型是否正确、置信度是否在 0 到 1 之间。

6. **异常处理**

   对超时、限流、模型返回非法 JSON、网络错误做明确处理。

### 1.5 面试表达

可以这样说：

> 我理解的 LLM 应用开发不是简单调模型接口，而是围绕模型调用构建完整工程链路。包括输入校验、Prompt 和上下文构造、模型 API 调用、结构化输出、结果校验、异常兜底和日志追踪。这样模型结果才能稳定被前端或后端业务系统消费。

### 1.6 对应到你的项目

在 Bug 分析 Agent 里：

- 用户输入是设备型号、固件版本、问题现象和日志。
- 上下文来自历史 Bug、模块文档和代码线索。
- 输出是 Bug 类型、根因、证据链和修复建议。
- FastAPI 和 Pydantic 负责接口和数据结构。
- 后续接入真实 LLM 时，可以把根因生成和报告生成节点替换成模型调用。

## 2. Prompt 设计、上下文构造、结构化输出和结果校验

### 2.1 Prompt 设计是什么

Prompt 设计是指如何给模型清晰描述任务、角色、约束和输出格式。

一个好的 Prompt 通常包含：

- 角色：你是一个嵌入式网通设备故障分析助手。
- 任务：根据日志和证据分析可能根因。
- 输入：设备型号、固件版本、现象、日志、历史 Bug、文档、代码片段。
- 约束：只能基于给定证据回答，证据不足要说明。
- 输出格式：返回 JSON，包含固定字段。

### 2.2 上下文构造是什么

上下文构造是指把哪些信息放进模型输入，以及按什么顺序组织。

例如 Bug 分析场景可以这样组织：

```text
任务说明：
你需要分析嵌入式网通设备故障。

用户输入：
设备型号：AX3000-GW
固件版本：v2.1.7
问题现象：升级后 DHCP 客户端无法获取 IP

日志片段：
...

历史 Bug：
BUG-018：DHCP server starts before bridge ready

模块文档：
DHCP 模块说明...

代码线索：
netifd_reload.c: restart_bridge 后立即 restart_dhcp_server

输出要求：
返回 JSON...
```

上下文构造的重点是：信息要足够，但不能堆太多无关内容。

### 2.3 结构化输出是什么

结构化输出就是让模型不要只返回一段自然语言，而是返回后端能解析的固定结构。

例如：

```json
{
  "bug_type": "network_dhcp",
  "summary": "DHCP 服务启动早于 LAN bridge ready",
  "root_causes": [
    "netifd reload 后 bridge 尚未进入 forwarding 状态，DHCP server 已启动"
  ],
  "evidence": [
    "log: dhcpd lease allocation failed",
    "bug: BUG-018"
  ],
  "fix_suggestions": [
    "等待 br-lan ready 后再启动 DHCP server"
  ],
  "confidence": 0.95
}
```

### 2.4 结果校验是什么

模型即使被要求输出 JSON，也可能输出错，比如：

- 少字段。
- 字段类型不对。
- JSON 格式非法。
- confidence 写成 `"high"` 而不是数字。
- bug_type 输出了不在枚举里的值。

所以需要用 Pydantic 或其他 schema 工具校验。

例如可以要求：

- `bug_type` 必须是字符串。
- `root_causes` 必须是字符串数组。
- `confidence` 必须是 0 到 1 之间的小数。

### 2.5 常见问题

#### 问题一：Prompt 越长越好吗？

不是。Prompt 太长会增加成本，也会让模型注意力分散。好的 Prompt 应该清晰、有结构、只放必要上下文。

#### 问题二：为什么模型不按格式输出？

可能是任务约束不清晰，也可能是上下文太复杂。可以通过更明确的 schema、few-shot 示例、JSON mode、工具调用或输出校验重试来改善。

#### 问题三：结构化输出会不会影响自然语言质量？

会有一点限制，但工程系统更需要稳定可解析。可以让模型先输出结构化字段，再由前端美化展示。

### 2.6 面试表达

可以这样说：

> 我会把 Prompt 分成任务说明、输入数据、检索证据和输出约束几个部分。对于后端需要消费的结果，我会要求模型返回结构化 JSON，并用 Pydantic 做校验。如果模型输出格式不符合要求，会进行重试或降级处理。

## 3. RAG 基本流程

### 3.1 RAG 是什么

RAG 是 Retrieval-Augmented Generation，中文通常叫检索增强生成。

它的核心思想是：

> 不让模型只依赖参数中的通用知识，而是在回答前先从外部知识库检索相关资料，再把资料作为上下文交给模型生成答案。

### 3.2 为什么需要 RAG

大模型有几个限制：

- 不知道企业内部私有资料。
- 知识可能过期。
- 容易幻觉。
- 无法天然引用来源。

RAG 可以解决这些问题：

- 把企业文档、历史 Bug、代码片段、SOP、FAQ 作为知识源。
- 知识更新不需要重新训练模型，只需要更新知识库。
- 回答可以绑定证据来源。
- 对专业问题更可靠。

### 3.3 RAG 离线流程

离线阶段主要是建知识库。

```text
收集知识源
  -> 清洗文档
  -> 文档切分
  -> 生成 Embedding
  -> 写入向量数据库
```

#### 文档清洗

去掉无用内容，比如：

- 重复页眉页脚。
- 空行和乱码。
- 无意义目录。
- 过期文档。

#### 文档切分

把长文档切成较小 chunk。

常见切分方式：

- 按标题切分。
- 按段落切分。
- 按固定 token 数切分。
- 代码按函数或类切分。
- 日志按时间窗口或错误上下文切分。

#### Embedding

Embedding 是把文本转成向量。语义相近的文本，向量距离更近。

#### 向量数据库

向量数据库用于存储 embedding 并做相似度检索。常见有 Chroma、FAISS、Milvus、Qdrant、pgvector 等。

### 3.4 RAG 在线流程

在线阶段是用户提问后的流程。

```text
用户问题
  -> Query 改写 / 关键词提取
  -> 检索相关 chunk
  -> rerank / 过滤
  -> 构造上下文
  -> 调用 LLM
  -> 生成带证据答案
```

### 3.5 chunk size 怎么选

chunk 太小：

- 检索精确，但上下文不完整。
- 可能把完整答案切碎。

chunk 太大：

- 上下文完整，但噪声多。
- 检索不够精确。
- token 成本更高。

一般可以从 300 到 800 tokens 开始，根据评估效果调整。

### 3.6 overlap 是什么

overlap 是相邻 chunk 之间保留一部分重复内容。

作用是避免重要信息刚好落在切分边界，被分到两个 chunk 里导致上下文断裂。

### 3.7 向量检索和关键词检索

向量检索适合语义相似问题。

例如：

```text
用户说：升级后客户端拿不到 IP
文档写：DHCP lease allocation failed after bridge reload
```

字面不完全一样，但语义相关，向量检索能召回。

关键词检索适合：

- 错误码。
- 函数名。
- 文件名。
- Bug ID。
- 日志关键字。

例如：

```text
BUG-018
lease allocation failed
restart_dhcp_server
```

实际项目里常用混合检索。

### 3.8 面试表达

可以这样说：

> 我理解 RAG 分为离线知识库构建和在线检索生成两部分。离线阶段对文档进行清洗、切分、embedding 并写入向量库；在线阶段根据用户问题检索相关 chunk，必要时做 rerank，再把检索结果作为上下文交给模型生成答案。RAG 的价值是让模型利用企业私有知识，并让回答有证据来源。

### 3.9 对应到你的项目

Bug 分析 Agent 中的知识源包括：

- 模块文档。
- 历史 Bug。
- 代码片段。
- 设备日志。

RAG 的作用是：

- 帮助系统找到类似历史问题。
- 给根因分析提供文档证据。
- 给修复建议提供代码线索。

## 4. Agent 应用设计

### 4.1 Agent 是什么

AI Agent 可以理解为一个能围绕目标进行多步骤执行的系统。

它通常包含：

- LLM：负责理解、推理和生成。
- Tools：外部工具，比如检索、数据库、API、脚本执行。
- State：任务状态和中间结果。
- Memory：历史信息或长期记忆。
- Planner / Controller：决定下一步做什么。

普通 ChatBot 更像单轮问答，而 Agent 更像任务执行系统。

### 4.2 Agent 和 Chain 的区别

Chain 通常流程固定：

```text
输入 -> 步骤 A -> 步骤 B -> 输出
```

Agent 更灵活：

```text
目标
  -> 判断需要什么工具
  -> 调用工具
  -> 观察结果
  -> 决定下一步
  -> 直到完成任务
```

不过在真实业务里，Agent 不一定要完全自由。很多时候会使用可控 Agent，也就是用状态机或图结构约束流程。

### 4.3 任务拆解

任务拆解是把一个复杂目标拆成多个步骤。

例如 Bug 分析可以拆成：

1. 提取故障信息。
2. 解析日志。
3. 判断 Bug 类型。
4. 检索历史 Bug。
5. 检索模块文档。
6. 检索代码线索。
7. 生成根因分析。
8. 输出修复建议。

拆解的好处：

- 每一步更容易测试。
- 出错时更容易定位。
- 每个节点可以单独优化。

### 4.4 工具调用

工具是 Agent 连接外部世界的方式。

常见工具包括：

- 文档检索工具。
- 数据库查询工具。
- 搜索工具。
- 代码执行工具。
- 企业系统 API。
- 工单系统 API。

设计工具时要注意：

- 工具名称要清楚。
- 参数 schema 要明确。
- 返回结果要结构化。
- 高风险工具要加权限和人工确认。

### 4.5 状态管理

Agent 多步骤执行时，需要保存中间状态。

例如：

```json
{
  "bug_type": "network_dhcp",
  "parsed_logs": {},
  "related_bugs": [],
  "related_docs": [],
  "related_code": [],
  "hypotheses": []
}
```

没有状态管理，复杂 Agent 会变得难以调试。

### 4.6 异常兜底

Agent 可能出现的问题：

- 工具调用失败。
- 检索为空。
- 模型输出格式错误。
- 任务超时。
- 多轮循环停不下来。

兜底方式：

- 设置 timeout。
- 设置最大重试次数。
- 对输出做 schema 校验。
- 检索为空时提示证据不足。
- 高风险动作要求人工确认。
- 记录每一步日志。

### 4.7 执行链路追踪

执行链路追踪是指记录 Agent 每一步做了什么。

例如：

```text
step 1: parse_logs -> 提取到 lease allocation failed
step 2: search_bug_history -> 命中 BUG-018
step 3: retrieve_docs -> 命中 dhcp.md
step 4: search_code -> 命中 netifd_reload.c
step 5: generate_report -> 输出根因
```

这样可以回答：

- 为什么 Agent 得出这个结论？
- 哪一步失败了？
- 检索有没有命中正确资料？
- 模型有没有使用证据？

### 4.8 面试表达

可以这样说：

> 我理解 Agent 应用的核心不是让模型随便行动，而是围绕任务目标进行可控的多步骤执行。设计时需要考虑任务拆解、工具调用、状态管理、异常兜底和执行链路追踪。实际业务中我更倾向于用 LangGraph 这类状态流框架约束 Agent 流程，让每一步都可观察、可测试。

## 5. LangChain / LangGraph 基础用法

### 5.1 LangChain 是什么

LangChain 是 LLM 应用开发框架，提供很多常用组件：

- LLM 调用封装。
- PromptTemplate。
- OutputParser。
- Retriever。
- Document Loader。
- VectorStore。
- Tool。
- Agent。

它的价值是让开发者更方便地把模型、知识库、工具和业务系统连接起来。

### 5.2 LangGraph 是什么

LangGraph 是 LangChain 生态中用于构建有状态、多步骤 Agent 的框架。

它的核心概念是：

- **State**：共享状态。
- **Node**：处理步骤。
- **Edge**：节点之间的执行关系。
- **Conditional Edge**：条件分支。
- **Graph**：完整工作流。

### 5.3 为什么用 LangGraph

复杂 Agent 通常不是简单线性流程，而是有状态、有分支、有循环、有失败恢复。

LangGraph 的优势：

- 流程清晰。
- 状态显式。
- 易于调试。
- 方便测试每个节点。
- 适合多步骤 Agent。

### 5.4 LangGraph 状态流示例

Bug 分析可以抽象成：

```text
START
  -> extract_bug_info
  -> parse_logs
  -> search_bug_history
  -> retrieve_related_docs
  -> search_codebase
  -> generate_hypotheses
  -> generate_report
  -> END
```

每个节点读取 state，并返回要更新的字段。

例如：

```python
def parse_logs_node(state):
    parsed = parse_syslog(state["logs"])
    return {"parsed_logs": parsed}
```

### 5.5 面试表达

可以这样说：

> LangChain 更偏 LLM 应用组件，比如 Prompt、Retriever、Tool、OutputParser；LangGraph 更适合编排有状态、多步骤 Agent。我在项目里会把复杂流程拆成多个节点，每个节点读写共享 state，这样比把所有逻辑塞进一个 prompt 更可控，也更方便测试和排查。

## 6. OpenAI-compatible API 接入方式

### 6.1 什么是 OpenAI-compatible API

很多模型服务会兼容 OpenAI 的 API 格式，比如：

- OpenAI。
- DeepSeek。
- Qwen。
- 部分私有化部署模型。

这样业务代码可以用类似格式调用不同模型。

常见请求结构：

```json
{
  "model": "xxx",
  "messages": [
    {"role": "system", "content": "你是一个助手"},
    {"role": "user", "content": "帮我分析这个日志"}
  ],
  "temperature": 0.2
}
```

### 6.2 接入时需要注意什么

#### API Key 管理

不要把 API Key 写死在代码里，应该放在环境变量或配置文件中。

#### Timeout

模型调用必须设置超时。

#### Retry

遇到临时网络错误、限流，可以重试，但要限制次数。

#### Model Config

模型名称、base_url、temperature、max_tokens 应该配置化。

#### 错误处理

要处理：

- 401 API Key 错误。
- 429 限流。
- 500 服务端错误。
- 网络超时。
- 模型返回空内容。

### 6.3 多模型切换

如果抽象得好，可以通过配置切换模型：

```text
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-chat
LLM_BASE_URL=https://api.deepseek.com
```

业务代码不需要大改。

### 6.4 面试表达

可以这样说：

> 我了解 OpenAI-compatible API 的接入方式。实际开发中会把 base_url、model、api_key、temperature 等参数配置化，模型调用设置 timeout 和 retry，并对返回内容做结构化解析和异常处理。这样后续可以在 OpenAI、DeepSeek、Qwen 或私有模型之间切换。

## 7. AI 应用常见工程问题

### 7.1 模型调用超时

模型调用可能因为网络、排队、上下文过长导致超时。

处理方式：

- 设置请求 timeout。
- 限制输入长度。
- 对长文档先摘要或检索。
- 超时后返回明确错误。
- 必要时使用异步任务。

### 7.2 输出格式不稳定

模型可能不按要求返回 JSON。

处理方式：

- 使用结构化输出。
- 使用 JSON mode 或 function calling。
- 增加格式示例。
- 用 Pydantic 校验。
- 校验失败时重试。

### 7.3 幻觉控制

幻觉是模型生成看似合理但没有依据的内容。

处理方式：

- 使用 RAG 提供证据。
- 要求答案引用证据。
- 证据不足时允许回答“不确定”。
- 降低 temperature。
- 对关键结论做证据一致性检查。

### 7.4 日志记录

AI 应用日志要记录：

- 请求 ID。
- 用户输入摘要。
- 检索 query。
- 命中的文档 ID。
- 模型名称。
- prompt 版本。
- 输出结果。
- 耗时。
- token 用量。
- 错误信息。

这些信息有助于排查问题和优化效果。

### 7.5 成本控制

模型调用成本主要来自 token 和调用次数。

控制方式：

- 用小模型处理简单任务。
- 缓存 embedding 和检索结果。
- 控制 TopK。
- 压缩上下文。
- 避免 Agent 无意义循环。
- 对长日志先规则提取，再交给模型。

### 7.6 安全与权限

AI 应用可能会连接内部系统，所以要考虑权限。

例如：

- 不能让模型自由执行任意命令。
- 工具调用要有白名单。
- 高风险操作需要人工确认。
- 敏感数据不能直接暴露给外部模型。
- API Key 要安全存储。

### 7.7 面试表达

可以这样说：

> AI 应用落地时会遇到很多工程问题，比如模型调用超时、输出格式不稳定、幻觉、日志追踪和成本控制。我的处理思路是：模型调用要有 timeout 和 retry，输出要结构化并校验，答案要尽量基于 RAG 证据，关键链路要记录日志，长上下文要压缩，工具调用要受权限和白名单约束。
